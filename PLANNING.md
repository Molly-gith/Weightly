# Weightly 项目规划

> 本文档是从「晨昏 · 体重日历」小程序项目的规划讨论中沉淀出来的,记录了 App+Web 这条产品线的所有已确认决策。新会话接手时,读完这份文档即可获得完整上下文,不需要重新讨论已经定过的部分。

## 项目背景与两条产品线的关系

「晨昏」体重管理产品目前拆成两条独立的线:

1. **小程序线**(仓库:`weight-tracker`,`Molly-gith/weight-tracker`)
   - 技术栈:uni-app(Vue3)+ uniCloud,一套代码出微信小程序 + H5
   - 现状:V1→V2→V3 的可交互原型 + 真实小程序代码已完成,**已冻结,不再迭代**
   - 保留原因:国内小程序使用场景仍然很多,不放弃这个入口

2. **App + Web 线**(仓库:`Weightly`,当前项目)
   - 全新起步,不复用小程序代码
   - 存在的意义:承载「AI Agent 产品」的长期野心,小程序生态工具链对这个方向支持有限

## 技术栈决策(已确认)

| 层 | 选型 | 理由 |
|---|---|---|
| 前端 | **Expo(React Native)+ Expo Router**,启用 `react-native-web` | 一套代码同时出 App(iOS/Android)和 Web,比"RN App + 单独 Next.js 网页"更彻底地复用代码 |
| 后端 | **Node.js + Express**,部署阿里云 **SAE**(Serverless 应用引擎) | AI 对话需要流式输出,纯函数计算(FC)在长连接/冷启动上体验打折扣,SAE 更合适;简单只读接口以后也可以下沉到 FC |
| 数据库 | **PostgreSQL**(阿里云 RDS for PostgreSQL),启用 `pgvector` 插件 | 用户数据(体重记录/目标)是标准关系型数据;`pgvector` 让"营养知识库 RAG 检索"直接复用同一个库,不用另外接向量数据库 |
| 仓库 | 独立新仓库 `Weightly`,Private | 两条线技术栈完全不同,分开仓库避免依赖/部署流程互相污染 |

## AI 助手产品讨论结论

**能力范围分三类:**
1. **数据类能力(工具调用)**:记录早晚体重、查看趋势——本质是"自然语言操作 App 已有数据",需要给 AI 挂 `record_weight`、`get_trend` 之类的工具函数
2. **情绪陪伴 + 建议(靠 prompt)**:先共情、再给可执行建议(比如"吃了华夫饼很懊悔"这类场景),技术门槛不高但内容质量要求高
3. **营养学知识 + 饮食记录**:⚠️ 这块和小程序一路坚持的"极简克制"是有张力的,需要单独决策,不要顺着 AI 功能丰富度的惯性就默认加上

**饮食记录范围已确认:** 只做减重方向的饮食/运动热量记录,**不做增肌**(受众更小,且减重和增肌需要的热量策略是矛盾的,增肌通常需要热量持平或盈余+高蛋白+抗阻训练,不是一个"每天差多少大卡"能同时满足两个目标的)

**知识库来源建议:**
- 食物热量/营养成分数据:参考《中国食物成分表》(中国疾控中心营养与健康所),或 USDA FoodData Central(免费 API,但中式菜品覆盖弱,作为补充)
- 营养学理论/减重计划类内容:**不建议让大模型自由生成**,建议自己整理一份精简知识库(可参考《中国居民膳食指南》改写),存成结构化短文档,用 RAG 检索后让模型引用着回答,内容口径可控可审查
- 健康建议类内容务必保留免责声明(小程序原型里已有"助手只提供陪伴与方向性建议,不替代医生或营养师",正式产品要保留甚至更显眼)

**架构选择:** 单 Agent + 可插拔工具的模式,不引入 LangGraph/多智能体编排框架——现在只有"体重"一个领域,没有编排的必要。工具之间边界要划清楚,方便以后真的往"生活助手"(记账/日记/复盘等)扩展时,能自然演化成"路由 Agent + 领域子 Agent",而不是推倒重来。

**不使用扣子(Coze)等无代码平台:** 后端本来就要读写自己的私有用户数据,无代码平台需要反向对接会增加不必要的复杂度和第三方依赖。直接用大模型官方的 Function Calling 接口自己实现 Agent Loop。

## Agent 实现的完整流程(供后续阶段参考)

1. 划定能力边界(能调用哪些工具、明确不能做什么——不能诊断、不能替代医嘱)
2. 系统提示词(persona)设计——语气、"先共情再给建议"的结构、免责声明时机
3. 工具函数实现(每个能力对应真实可调用的后端函数)
4. 对话记忆管理(保留多少轮上下文、要不要存长期记忆)
5. 知识库检索接入(RAG)
6. 编排循环(用户消息 → 带工具定义调用大模型 → 按需执行工具 → 结果喂回 → 循环至最终回复)
7. 护栏(内容审核、敏感话题拒答策略)
8. 测试评估(用真实场景做测试用例,上线前人工审一遍)
9. 前端接入(聊天界面,考虑流式输出)
10. 上线迭代(监控 token 成本和真实对话效果)

## 分阶段路线图

| 阶段 | 内容 | 依赖 |
|---|---|---|
| 1. 项目脚手架 | 新仓库 + Expo 项目 + Express 后端骨架 + 打通 PostgreSQL 连接,健康检查接口跑通全链路 | 无 |
| 2. 核心记录功能复刻 | 今天页(晨重/晚重/减重旅程)+ 日历页(点格即录、趋势图),复刻小程序 V3 体验,但要分别设计 App 和 Web 布局(Web 屏幕更宽,可以考虑侧边栏导航、今天+日历并排) | 阶段 1 |
| 3. 账号系统 | 登录注册(App+Web 通用,不依赖微信身份) | 阶段 1 |
| 4. AI 助手基础版 | 工具调用(记体重/查趋势)+ persona + 对话界面 | **需要用户提供大模型 API Key** |
| 5. 营养知识库 + 建议能力 | 整理知识库内容 + RAG 检索接入 | 阶段 4 |
| 6. 饮食热量记录(仅减重方向) | 新数据模型 + 界面 + 热量计算工具 | 阶段 2、5 |
| 7. 多智能体拓展 | 往"生活助手"方向扩展(记账/日记/复盘等),远期方向,现在不排期 | 待第二个领域出现后再规划 |

- ✅ 仓库 `Weightly` 已创建(Private)
- 🔲 **正在设计 V1 原型**(交互式 HTML 原型,和小程序线的做法一致,先设计后写真代码)
